# FileOps - 产品需求文档（PRD）

> 版本：v0.1（草案）  
> 日期：2026-03-23  
> 适用形态：CLI 工具 + 可被其他程序调用的库（SDK）

## 1. 背景与问题
在日常运维、数据处理、内容生产、研发协作中，经常存在“文件级”的批量处理需求（搬运、改名、归档、去重、校验、清理、同步等）。常见痛点：
- 手写脚本一次性、不可复用：缺少统一配置、缺少审计与回滚、错误处理不一致。
- GUI 工具难自动化：难纳入 CI/定时任务，难在多环境复现。
- 批量操作高风险：误删/误覆盖、跨盘移动失败、编码/权限问题、符号链接/路径穿越等安全问题。

FileOps 的定位是：**“可配置、可预演（dry-run）、可审计、可扩展”的文件操作执行引擎**。

## 2. 目标与非目标
### 2.1 目标（Goals）
- 提供稳定的文件扫描/过滤能力，把“要处理哪些文件”描述清楚、可复现。
- 提供常用文件操作，并且支持“先生成执行计划，再执行”的安全模型。
- 支持 dry-run、确认提示、可恢复执行、输出报告，降低批量操作风险。
- 支持插件化扩展（新增操作/过滤器/报告导出）。
- 跨平台（Windows/macOS/Linux）可用，尤其对 Windows 路径/编码友好。

### 2.2 非目标（Non-goals，v0.x 不做）
- 不做“完整文件管理器/网盘客户端/图形界面”（可后续扩展为 GUI Wrapper）。
- 不做“全量备份系统/分布式存储”（可提供与外部系统对接的接口）。
- 不承诺替代 `rsync` 等专用同步工具的全部能力（v0.x 聚焦通用场景）。

## 3. 目标用户与场景
### 3.1 用户画像
- 研发/数据/运维：需要可在脚本、CI、定时任务中稳定运行的批处理工具。
- 内容生产/设计：需要批量重命名、归档、格式转换（可由插件提供）。
- 安全/合规：关注可审计、可追溯、可回滚（至少可“生成变更清单”）。

### 3.2 典型场景（示例）
- 依据规则把下载目录按日期/类型归档，并保留可追溯报告。
- 扫描大文件或重复文件，生成报告并选择性清理到回收站。
- 按模板批量重命名图片/音频，避免重名覆盖，保留映射表。
- 对目录做“同步/镜像”（基础版）：复制缺失、校验哈希、输出差异清单。

## 4. 核心概念与术语
- **Workspace**：一次运行的根目录/允许操作的路径范围（安全边界）。
- **Scanner**：扫描器，从文件系统枚举候选条目（文件/目录/链接）。
- **Filter/Rule**：过滤规则（glob/正则/大小/时间/扩展名/哈希等）。
- **Operation**：对条目执行的动作（copy/move/rename/delete/...）。
- **Plan**：执行计划（将“将要执行的操作”结构化列出，可审查、可导出）。
- **Executor**：执行引擎，按计划执行并产出结果（成功/跳过/失败原因）。
- **Report**：运行报告（汇总 + 明细 + 可选导出格式）。
- **Plugin**：扩展点，新增 operation/filter/reporter。

## 5. 功能需求（Functional Requirements）
### 5.1 CLI 基础
- 支持 `fileops <command> [options]` 形态。
- 通用选项：
  - `--config <path>`：加载配置文件（建议 YAML/TOML/JSON 三选一，文档默认以 YAML 表达）。
  - `--dry-run`：仅生成计划不落盘修改。
  - `--yes`：非交互确认（适配 CI/定时任务）。
  - `--log-level`：日志级别；`--log-json` 输出结构化日志（便于采集）。
  - `--report <path>`：输出报告（JSON/CSV/HTML 至少支持一种）。

### 5.2 扫描与过滤
- 支持从一个或多个根目录扫描：
  - 递归深度限制、是否跟随符号链接、是否包含隐藏文件。
  - 忽略列表（.gitignore 风格可选）。
- 过滤条件（可组合 AND/OR）：
  - 路径匹配：glob/正则
  - 类型：文件/目录/链接
  - 扩展名/文件名模式
  - 大小范围、修改时间范围
  - 内容特征（可选）：文本包含、MIME/魔数探测（插件化）
  - 哈希（可选）：用于去重/校验（成本高，应可选开启）

### 5.3 常用操作（v0.x 目标集）
- `copy`：复制到目标目录；支持保留目录结构；支持覆盖策略（禁止/覆盖/改名）。
- `move`：移动到目标目录；跨盘移动需要回退到 copy+delete；失败可恢复。
- `rename`：按模板/规则重命名：
  - 模板变量：`{stem}` `{ext}` `{index}` `{date}` `{parent}` `{hash8}` 等
  - 冲突策略：自动追加序号、跳过、报错
  - 输出映射表（旧->新）
- `delete`：删除策略：
  - 优先“移入回收站/Trash”（跨平台实现可能不同）
  - 可配置“硬删除”但必须二次确认或显式 `--force`
- `mkdir` / `touch`（可选）
- `dedupe`（可选，v0.2+）：基于大小+哈希判定重复，保留策略（最新/最早/路径优先）。
- `sync`（可选，v0.2+）：源到目标的基础同步，差异清单 + 执行。

### 5.4 执行模型与安全护栏
- **Plan-first**：任何会改动文件系统的命令应先构建 `Plan`：
  - 展示摘要（影响文件数/预计写入字节/潜在覆盖数）
  - 支持导出计划文件，供审核/复跑
- **Dry-run**：默认支持所有命令；dry-run 不写盘（但允许读取元数据/计算可选哈希）。
- **Workspace 约束**：
  - 所有操作必须位于允许路径范围内（防止误操作系统目录）。
  - 禁止路径穿越（`..`）、危险路径（根目录）等；规则可配置。
- **原子性与可恢复**（渐进实现）：
  - 为每个操作记录结果与中间状态，支持 `--resume` 继续未完成项。
  - 关键操作（move/rename）尽量使用原子 rename；跨设备时记录临时状态。
- **并发**：
  - I/O 密集型操作支持并发（可配置 worker 数），但要保证日志与报告一致性。

### 5.5 日志与报告
- 运行时日志：
  - 每条操作记录：输入、输出、耗时、结果码、错误信息（结构化字段）。
- 报告输出：
  - 汇总：扫描数、命中数、成功/失败/跳过、耗时、吞吐
  - 明细：每个条目的操作结果（可用于审计与回滚脚本）
  - 导出格式：JSON 必须；CSV/HTML 可选

### 5.6 插件机制（v0.2+）
- 插件能力：
  - 新增 Operation（例如：压缩、转码、上传、OCR）
  - 新增 Filter（例如：按 EXIF、按内容相似度）
  - 新增 Reporter（例如：写入数据库、发送通知）
- 插件发现：
  - 本地目录加载（`plugins/`）
  - 或包管理 entrypoints（若采用 Python 等生态）

## 6. 非功能需求（Non-Functional Requirements）
- **正确性**：绝不默默覆盖/删除；默认安全优先（dry-run + 明确确认）。
- **可用性**：CLI 输出可读，提供示例配置与常见命令模板。
- **可观测性**：结构化日志；错误分级；可输出机器可读报告。
- **性能**：在 10 万文件枚举下仍可接受（扫描与过滤需流式处理）。
- **兼容性**：Windows 路径、长路径、Unicode 文件名、权限/锁定文件要有明确策略。
- **可测试性**：核心逻辑与 I/O 解耦；可用临时目录做集成测试。

## 7. 权限与安全（补充）
- 默认不跟随符号链接（可配置开启）。
- 默认禁止对系统敏感目录执行写操作（可配置允许列表）。
- 对 delete/hard-delete、覆盖写入等高风险操作要求额外确认或 `--force`。

## 8. 优先级（MoSCoW）
### Must（v0.1）
- 扫描/过滤（glob + 基础属性）
- copy/move/rename/delete（回收站优先）
- plan-first + dry-run + 确认提示
- JSON 报告输出
- 结构化日志（至少可选）

### Should（v0.2）
- 去重（大小+哈希）
- 基础同步（差异清单）
- 插件机制（Operation + Reporter）
- `--resume` 恢复执行

### Could（v0.3+）
- GUI Wrapper
- 规则 DSL/可视化规则编辑
- 与对象存储/网盘/消息通知集成（插件）

### Won’t（短期不做）
- 分布式/多机协调执行
- 强一致事务型文件系统抽象

## 9. 里程碑建议
- **v0.1（MVP）**：copy/move/rename/delete + 扫描过滤 + plan/dry-run + JSON 报告
- **v0.2**：去重/同步（基础）+ 插件机制 + 恢复执行
- **v0.3**：扩展插件生态 + 更多报告/通知渠道 + 性能优化

## 10. 未决问题（需要进一步澄清）
1) 主要运行环境是否以 Windows 为主？是否需要兼容网络盘/UNC？  
2) 配置格式偏好（YAML/TOML/JSON）与团队既有工具链？  
3) “删除”是否必须统一走回收站？是否允许硬删除？合规要求是什么？  
4) 需要哪些内置操作的优先级（压缩/解压/校验/改权限/文本替换等）？  
5) 预期最大数据量与性能指标（文件数/总容量/运行时长）？
